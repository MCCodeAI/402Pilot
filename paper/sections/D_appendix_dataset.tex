\section{402Pilot-Bench Artifact Details}
\label{appendix:dataset}

The main text defines the benchmark task buckets, providers, market
scenarios, metrics, and limitations. This appendix records only the
artifact-level details needed to inspect or rerun 402Pilot-Bench:
source sampling, cache layout, score persistence, and maintenance
conventions.

\subsection{Source sampling and cache layout}
\label{app:dataset-sampling}

402Pilot-Bench draws tasks from four public sources. HumanEval contributes
all $164$ canonical tasks~\citep{chen2021humaneval}. The HotpotQA
validation split~\citep{yang2018hotpotqa},
TriviaQA-web~\citep{joshi2017triviaqa}, and OpenAssistant
oasst1~\citep{kopf2023openassistant} subsets are sampled with
NumPy-default RNG using loader-level constants. The sampled indices are
sorted and recorded in per-source cache files so the task pool can be
reconstructed without depending on upstream dataset order. T3b prompts in
\texttt{data/tasks/openweb.jsonl} retain their \texttt{OpenAssistant/oasst1}
source tag and original oasst1 \texttt{message\_id}. For T3a, one
TriviaQA-web item is removed at load time by a global content-filter
skip-list; the on-disk cache keeps the raw $220$-task sample, and removing
the skip-list entry restores it without rebuilding the cache.

For each (task, provider) pair we pre-generate $V = 5$ response versions,
stored as JSONL files, one file per (provider, bucket) pair. Each record
stores the schema version, task identifier and bucket, provider identifier,
response-version index, frozen response text, realized cost and latency,
failure flag and code, quality score, generation timestamp, and sampling
temperature. Quality is materialized at pre-generation time: deterministic
scorers write their scores directly, while T3b stores cached LLM-as-judge
outputs in \texttt{judge\_cache.jsonl} with the fixed rubric and provenance
needed to audit the judge call.
The run configuration file lists provider menus, behavioral wrappers, and
scenario schedules; in the artifact, this file is
\path{experiments/main.yaml}. Wrapper randomness is keyed by the per-call
seed so replay results are deterministic.

\subsection{Reproducing tables}
\label{app:dataset-reproducing-tables}

The released artifact separates replay generation from table
aggregation. Recomputing tables from existing JSONL logs requires no live
LLM calls and usually takes minutes on a laptop; rerunning the full
$30$-seed, $10{,}000$-round sweeps can take several CPU-hours and is
dominated by local JSONL I/O.
The artifact README documents the one-shot commands for recomputing
Table~\ref{tab:main}, the paired significance checks, Table~\ref{tab:ablations},
and the S3 cost-posterior diagnostic from either cached logs or regenerated
replay runs.

\subsection{Distribution and maintenance}
\label{app:dataset-distribution}

The pre-generated responses, the loader code, the scenario classes, the
configuration YAML, and the scripts that regenerate every table and
figure in the paper are released as the supplemental artifact. Bug fixes
that change the on-disk cache will bump a \texttt{schema\_version} field;
fixes that change only load-time filters, such as the content-filter skip
list, leave the cache file untouched.
