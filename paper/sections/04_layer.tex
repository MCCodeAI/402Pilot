\section{The 402Pilot Decision Layer}
\label{sec:layer}

402Pilot is a decision layer between an autonomous agent and a programmable payment protocol such as x402. The layer accepts a task and wallet state, outputs a provider choice, asks the protocol stack to execute the selected payment, and feeds the realized outcome back into a policy. It does not modify the protocol, never holds funds, and never crosses the settlement boundary. Figure~\ref{fig:architecture} sketches the per-round flow.

The architectural contract is intentionally small. A policy is any object exposing
\begin{align*}
&\mathtt{select}(\mathbf{x}_t,\, \mathcal{A}_t) \to a_t \in \mathcal{A}_t, \\
&\mathtt{update}(\mathbf{x}_t,\, a_t,\, \ut,\, c_t) \to \emptyset,
\end{align*}
where $\mathcal{A}_t \subseteq \{1, \ldots, K\}$ is the set of providers whose nominal listed prices fit in the remaining wallet. The candidate provider set is supplied by an external discovery or registry component; 402Pilot only filters it for wallet feasibility. The policy receives the feasible set, but numeric effective charges for unchosen providers are not passed to it. Wallet pressure is an optional dependency: PA-DCT reads $\lambda_t$ from the wallet inside \texttt{select}, while non-payment-aware comparators carry no wallet dependency. The two methods are therefore the only points of contact with the rest of the stack; priors, posteriors, and other internal state remain private to the policy.

A key design choice is that the layer feeds \emph{utility} $\ut$ to \texttt{policy.update}, rather than the payment-aware reward $\rt$. The same payment-aware form defines the decision-time ranking score and the evaluation objective, but realized $\rt$ should not be treated as an observation of provider quality. Updating posteriors on $\rt$ would make the same provider appear worse late in a budget-tight run than early in the run, even if its service behavior had not changed. In 402Pilot, beliefs about provider utility and cost are updated from $(\ut,c_t)$, while wallet pressure enters selection through $\lambda_t$.

The same contract gives the layer two forms of pluggability. On the policy side, every comparator we evaluate---Random, the three Always-X variants, BudgetRule, and PA-DCT itself---implements the same two methods, so algorithms can be exchanged without changing the surrounding runtime. On the protocol side, although we instantiate the layer above x402~\citep{coinbase2025x402v2}, the contract applies to any pay-per-call protocol that exposes listed prices, settles selected calls, and returns payment receipts. The protocol stack provides addressing, settlement, and audit trails; 402Pilot supplies the buyer-side decision rule before settlement is invoked.

This boundary also fixes what 402Pilot does not attempt to solve. It does not implement settlement, provider discovery, production observability, or multi-agent coordination; those concerns belong to the protocol, registry, operator, or higher-level orchestration layers. We study a single-agent setting with one wallet, one policy, and one workload stream.

The boundary relies on three assumptions about the surrounding stack. First, the protocol or registry exposes nominal listed prices before paid calls, allowing the wallet to compute feasibility; realized effective cost is logged only for the selected call. In our benchmark, listed prices upper-bound realized charges. Second, outcomes are observable and scoreable shortly after the call, so the selected provider can be updated before the next round; in our benchmark, this is implemented with deterministic task evaluators or cached LLM-as-judge scores. Third, providers are non-strategic in the mechanism-design sense: they post prices and deliver services rather than bidding strategically for allocation. Strategic providers, as studied in recent mechanism-design work~\citep{patra2026reverseauction}, are complementary to this layer rather than a substitute for it.

Our reference implementation is a Python package (\texttt{pilot402}). The policy interface, wallet, context encoder, reward calculator, and per-seed runtime loop are separate modules, so each can be replaced without changing the others.
